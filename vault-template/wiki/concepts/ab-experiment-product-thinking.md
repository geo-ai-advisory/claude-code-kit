# Product thinking для А/Б экспериментов showcase

## TL;DR (как думать продактом, не разработчиком)

Цель А/Б тестов витрины — **генерить деньги через правильное размещение каждого оффера**. Не «сравнивать офферы между собой», а **«определять лучшее место для каждого оффера»** и **автоматически собирать гипотезу для теста**.

## Корневая ошибка — сравнение absolute EPC между офферами

Heatmap «оффер × позиция → EPC» с цветом по абсолютному значению — **бесполезен** для product решения. Почему:

- BestStandard платит 250 ₽ за выдачу, ДеньгиСразу 50 ₽ → BestStandard всегда «зеленее» в любой позиции
- Heatmap превращается в визуализацию **ставок партнёров**, а не в визуализацию **«где какой оффер работает лучше»**
- Пользователь смотрит и думает «ну BestStandard лучший, поставлю его везде» — это уже видно без heatmap

## Правильный взгляд — оффер-центрический

Для каждого оффера отдельно: **в какой позиции (1-7) он даёт лучший EPC/CR относительно своих же показателей в других позициях**.

Например:
- BestStandard: позиция 1 → EPC 280, позиция 7 → EPC 220 → «хочет позицию 1»
- ДеньгиСразу: позиция 1 → EPC 40, позиция 7 → EPC 90 → «хочет позицию 7» (anchor effect, см. [[wiki/concepts/showcase-anchor-position]])
- Webbank: позиция 3 → EPC 180, позиция 5 → EPC 100 → «хочет позицию 3»

Из этих «хотелок» собирается **гипотеза порядка витрины** — для каждой позиции выбирается оффер, который в этой позиции максимизирует ожидаемый EPC.

## Визуализация — нормированный heatmap

Heatmap должен показывать НЕ absolute EPC, а **относительный rank внутри строки оффера**:

- Каждая строка = один оффер
- Цвет ячейки = ранг этой позиции для этого оффера (1=лучшая позиция, N=худшая)
- Цветовая шкала отдельна для каждой строки (нормировка row-wise)
- Сверху можно опционально показать absolute EPC цифрой, но цвет = относительный

Альтернатива: показать **delta EPC vs avg оффера** — насколько эта позиция лучше/хуже среднего для оффера.

## Auto-hypothesis на лидерборде

На странице лидерборда (под текущей таблицей вариантов) — **автогенерируемая гипотеза**:

| Позиция | Оффер | Обоснование |
|---|---|---|
| 1 | BestStandard | EPC 280 в позиции 1 vs 220 в позиции 7 |
| 2 | Webbank | EPC 180 в позиции 3 (best для Webbank) |
| ... | ... | ... |
| 7 | ДеньгиСразу | EPC 90 в позиции 7 (anchor) |

С кнопкой **«Протестировать как новую гипотезу»** — создаёт новый вариант в текущем эксперименте с этим порядком. Не «применить как базовый», а добавить в pool вариантов на test.

Это превращает страницу из observe-tool в action-tool: пользователь видит данные → одна кнопка → новый вариант в работе → деньги.

## Контракт для product-architect (любая UI-задача про А/Б)

Перед тем как делать screen-spec для любой страницы экспериментов / heatmap / лидерборда — задать дополнительно к 7 базовым вопросам:

1. **Какое решение пользователь должен принять после увиденного экрана?** (если ответ «никакое» — экран бесполезен)
2. **Какая action-button должна быть на экране?** (если нет — observe-only, нужно ещё подумать)
3. **Что генерит деньги из этих данных?** (правильный порядок витрины → больше выдач → больше КВ → деньги)
4. **Сравниваем ли мы вещи которые сравнимы?** (absolute EPC между офферами с разными ставками — несравнимы; делать row-normalized)
5. **Есть ли auto-hypothesis который можно показать пользователю?** (если данных достаточно — система сама должна предложить «попробуй вот так»)

## Эксперимент — живой процесс, не разовый contest

Корневая ментальная ошибка разработчика: «эксперимент = вариант A vs вариант B → собрали статистику → ОДИН промот → конец».

Реальность продакта-овнера:
- Базовый вариант **переключается на ходу** (хоть 10 раз в день) на основе данных, гипотез, рынка, чутья
- При переключении эксперимент **продолжается**, просто меняется control
- Старый базовый **остаётся в истории** с пометкой «был базовым с такого-то по такое-то»
- Эксперимент закрывается **отдельным действием**, не вместе с промотом

### Mental model — версионируемый базовый

```
Эксперимент = бесконечный процесс
  ├─ Текущий базовый: variant X (с 2026-05-10 14:00)
  ├─ История базовых:
  │   ├─ variant Y был базовым 2026-05-08 → 2026-05-10
  │   └─ variant Z был базовым 2026-05-05 → 2026-05-08
  ├─ Активные варианты на тесте: A, B, C, D, E, F
  └─ Закрытие эксперимента — отдельная кнопка, не связана с промотом
```

### UX следствия для любого экрана А/Б эксперимента

- Кнопка «Сделать базовым» НЕ должна закрывать эксперимент. Только меняет control variant
- При промоте — toast «Базовый изменён. Старый базовый сохранён в истории» с undo (5 сек)
- Отдельная секция «История базовых» на странице эксперимента — таблица с диапазонами дат
- Confirm-диалог: «Сменить базовый на X? Эксперимент продолжается, старый базовый Y будет помечен как ‘был с DD.MM HH:MM по DD.MM HH:MM’»
- Закрытие эксперимента — отдельная кнопка в админ-зоне с другим визуалом (destructive action)

## Анти-паттерны (что НЕ делать)

- ❌ Сравнивать absolute EPC между офферами без нормировки
- ❌ Делать heatmap «информативным», без actionable вывода
- ❌ Прятать кнопку «сделать базовым / протестировать» если данных «недостаточно» — оставить с warning «принудительно, 0 показов»
- ❌ Скрывать варианты с малым числом кликов — пользователь хочет видеть теоретические победители
- ❌ Слетевший cursor в селекторе при typing
- ❌ Решать «как разработчик» («тут 0 кликов → ничего показывать») вместо «как продакт» («у меня есть гипотеза → дай мне кнопку чтобы её протестировать»)
- ❌ **Промот варианта = закрыть эксперимент.** Это разные действия, не связывать
- ❌ Терять историю предыдущих базовых при промоте
- ❌ Делать confirm-диалоги в стиле «final destruction warning» когда это рутинная операция
- ❌ Показывать «★ базовый» без полного порядка офферов — пользователь не экстрасенс, он не помнит порядок наизусть
- ❌ Рендерить **одну и ту же entity** (порядок офферов) по-разному в разных секциях одной страницы (см. [[wiki/concepts/component-reuse-discipline]])
- ❌ **Промот варианта = закрыть эксперимент.** Это разные действия, не связывать.
- ❌ Терять историю предыдущих базовых при промоте
- ❌ Делать confirm-диалоги в стиле «final destruction warning» когда это рутинная операция
