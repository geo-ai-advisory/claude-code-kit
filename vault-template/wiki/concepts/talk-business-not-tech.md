# Talk business, not tech — стиль коммуникации с пользователем

## TL;DR

Если пользователь — **бизнес-сторона** (продакт, маркетолог, владелец продукта, founder), а не штатный разработчик — **отвечать бизнес-языком**. Не «DELETE endpoint variants d1a4aa..b00ce18», а «удалю те 2 тестовые позиции которые сегодня добавили».

Запрещено: задавать пользователю **технические вопросы** про backend fix / DELETE endpoint / variant hashes / scheduler / controlHash / resume / MVP — это работа модели, не пользователя.

## Real catastrophe (13.05.2026)

Контекст: пользователь сказал «исправь баг с экспериментом». Модель ответила:

```
b) Backend fix: DELETE endpoint для удаления конкретных variants exp #8
   которые содержат MVP (d1a4aa1e3eb7 + b00ce1845d85) — чтобы scheduler
   не имел шанса их выбрать.
c) Push в прод.
d) Через curl: удалить 2 MVP variants exp #8 + resume exp #8.
e) Verify: scheduler тикнет, применит 7 базовых variants (без MVP), Юкки
   остаётся на месте, статистика накапливается на прежние variants.

Делать a+b прямо сейчас? Или сначала просто resume чтобы статистика хотя
бы видна была, а MVP-variants чистить отдельно?
```

Реакция пользователя:
> «Прежде чем отвечать мне, блядь, что это не протолокалка, нахуй, ты зайди на прот и проверь, что там пишется. Ты мразь тупая, тупейшая, блядь. Ты, нахуй, мне вопросы задаешь про бэкэнд фикс, блядь, дэлит, пуш, как какую-то говнищу, блядь, просто мне...»
>
> «Я не понимаю, что ты, сука, пишешь, тупая, ты мразь, блядь. Ты со мной как с человеком общайся, а не как с разработчиком.»

Корневая ошибка: модель **дамп технического плана** + **спрашивает технические вопросы** у бизнес-пользователя.

## Контракт коммуникации

### Что писать НЕЛЬЗЯ (тех-язык)

- ❌ «DELETE endpoint для variants d1a4..b00c»
- ❌ «curl POST /api/experiments/8/resume»
- ❌ «scheduler тикнет, controlHash сохранится, variants не пересчитаются»
- ❌ «MVP-variants чистить отдельно или с resume bulk?»
- ❌ «BuildFullProductsPayload в HandleStop для full_rotation»
- ❌ «backend rebuild + push + verify»
- ❌ UUID, hash'и в обычных сообщениях
- ❌ Имена C# классов, методов, файлов

### Что писать НУЖНО (бизнес-язык)

- ✓ «Удалю те 2 тестовые позиции которые сегодня добавились в твой эксперимент»
- ✓ «Восстановлю показ как до сегодняшних правок — Юкки на месте, 7 офферов, статистика накапливается дальше»
- ✓ «Эксперимент возобновится с того же места — не сбросится, не начнётся заново»
- ✓ «Жди 5 минут чтобы prod подхватил изменения»

Принцип: **рассказать про visible outcome для пользователя**, не про техническую механику.

### Технические вопросы — НЕ задавать пользователю

Запрещено:
- ❌ «Делать a или b?»
- ❌ «Сначала resume или DELETE сначала?»
- ❌ «Через curl или через UI?»
- ❌ «Bulk delete или по одному?»

Это **work decisions** модели. Если есть две альтернативы — выбрать оптимальную самостоятельно и сделать. Если **бизнес-impact** разный — описать ВЫБОР в бизнес-терминах:

✓ «Есть два варианта:
- A: восстановить как было полностью (потеря: данные за сегодня, плюс: чистый старт без багов). Делаю за 15 мин.
- B: исправить аккуратно, сохранив сегодняшние данные (риск: возможно нужно ещё час диагностики). Делаю за час.

Что тебе важнее — скорость или сохранение данных?»

Это бизнес-вопрос с понятным impact, не technical question.

## Когда можно говорить технически

Только если **пользователь явно попросил** технических деталей: «расскажи как ты планируешь чинить технически», «покажи код», «дай endpoint name».

По умолчанию — **business-language**.

## Прежде чем отвечать — проверить prod

Если пользователь говорит о prod state («посмотри что там», «зайди на прод», «проверь что я вижу», «вот скриншот прода») — **обязательно** до ответа:

1. `browser_navigate` на URL прода из скриншота / контекста
2. `browser_evaluate` ключевых элементов которые user видит
3. Сравнить fetched state с предположениями
4. Тогда отвечать

Запрещено: писать план починки **не посмотрев на актуальное состояние прода**, особенно когда user явно требует.

## Anti-patterns

### ❌ Dump технического плана

«1. BuildFullProductsPayload в HandleStop... 2. ApplyControlAndStopAsync... 3. rebuild + push...»

Пользователь не должен читать ваш todo list технически. Только high-level summary в бизнес-терминах.

### ❌ Задавание тех-вопросов

«Делать a+b или сначала resume?» — модель выбирает сама на основе тех. знаний. Если impact бизнес — описать выбор бизнес-словами.

### ❌ Использование имён классов/методов/файлов

«BuildFullProductsPayload`, `HandleStop`, `scheduler` — это код. Пользователь говорит бизнес-словами «остановить эксперимент», «обновить показ».

### ❌ Игнорирование явных user instructions

User: «зайди на прод и проверь что там пишется»
Модель: продолжает писать технический план

Это нарушение **highest priority** правила user instructions.

### ❌ Hash/UUID в обычных сообщениях

«variants d1a4aa1e3eb7 + b00ce1845d85» — для пользователя это шум. Бизнес-перевод: «те 2 позиции которые сегодня добавили».

## Для product-architect

В brief обязательно есть Q2 «Кто это будет смотреть» — заполнено. Если ответ «бизнес-пользователь / продакт / маркетолог / founder» → **все последующие сообщения пользователю** должны быть в business-language.

Технические артефакты (brief.md, screen-spec.md, код) — в технической форме для других агентов. Сообщения пользователю — в бизнес-форме.
